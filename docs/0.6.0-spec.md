# 0.6.0 Implementation Spec — The Location Wizard

> **Companion to:** `docs/0.6.0-design.md` (the *why*). This document is the *how*:
> the exact crates, files, types, and call sites to change, in the build order the
> design locked (§6), with acceptance criteria per increment.
>
> **Read first:** `docs/architecture.md` §4 (Wizard Framework) + §8D, and
> `docs/command-contexts.md` §1/§4. The wizard engine (`wizard` crate) and the
> dungeon reference (`desktop/src-tauri/src/wizards/dungeon.rs`) are the templates
> this spec leans on. Nothing here invents new engine machinery.

---

## 0. Summary of the approach

`create location` becomes a guided **wizard** registered on the existing engine,
exactly mirroring `create dungeon`:

- **One new file**, `desktop/src-tauri/src/wizards/location.rs`, holds the whole
  wizard (accumulator + steps + `Wizard` impl + `finalize`).
- **One registry line** in `wizards/mod.rs`.
- **One launch-routing change** in `commands/create_commands.rs`.
- The wizard's `finalize()` calls `editor.set_location(draft)` — the *same*
  hand-off the current one-shot `create_location` performs — so `LocationDraft`,
  `location_entity_card`, and the entire `location show|rename|set|reroll|save|cancel`
  surface keep working unchanged (architecture §4 "a kind can be wizard-created
  *and* entity-edited"; this is the dungeon model applied to location).
- A **kind-aware generation method** on `AiGenerationService` produces only the
  LLM-derived fields, under the GM's locked answers, with a per-kind JSON schema.

The wizard id is **`"location"`** — the same string as the entity command root.
This is deliberate and safe: the live context is `InputContext::Wizard("location")`
while the wizard runs and flips to `InputContext::EntityEditor("location")` after
`finalize()` opens the draft. They are distinct enum variants, never active at
once (command-contexts §1). Dungeon already does exactly this.

### Decisions made in this spec (and why)

| Question the design left to implementation | Decision | Rationale |
| --- | --- | --- |
| How does a flat step list branch by kind? | Step 1 (`kind`) returns `WizardTransition::Goto(<branch entry step>)`; each branch's **last** step calls generation and `Goto("review")`. `Next` is used only *within* a contiguous branch. | The engine has no built-in branching; `Goto(step_id)` is the mechanism (`wizard/src/runtime.rs`). Mirrors dungeon's single `Goto("plan_review")`. |
| Where does generation fire? | In the **last (geography / custom-seed) step's `accept()`**, via a shared `generate_location_into(data, state)` helper, then `Goto("review")`. `finalize()` does **no** LLM call — it just builds the draft from the already-generated seed + locked answers. | One pass is enough for a location (unlike dungeon's two-pass story→structure). Generating before `review` is what lets `review` show the card. Mirrors dungeon's `generate_story` helper shared across steps. |
| `create location <prompt>` fast path | Stays the **existing one-shot `create_location` handler** (unchanged). Only **bare** `create location` launches the wizard. | Design §3: "the custom branch (and `create location <prompt>`) is just the old one-shot path." The one-shot already does "model's call for every field" = the Custom column of §5. Routing the `<prompt>` form through the wizard would require an engine change to `start_wizard` (it always renders step 0); not worth it. |
| `exports` "derived, not invented" | Settlement generation feeds Q-B (resources) + Q-C (export mode) into the prompt and asks the model to produce a 1–3 item `exports` list *consistent with* them (raw→ship the resource, refined→the processed good). Site/Hideout: `exports` is omitted from the schema entirely and stored as an empty `Vec`. | Raw→refined needs world knowledge (grain→flour), so it's an LLM field — but now *grounded* by the locked inputs rather than hallucinated from nothing. Suppression is schema-level so the model never emits it. |
| `danger_level` ownership | Settlement: LLM-derived (in schema). Site/Hideout: **GM-locked** — omitted from the schema, injected from the GM's Q-S2/Q-H3 answer in post-processing. | Design §2/§5: for a found site danger is the point. |

## 1. Data model & contracts (no new draft fields)

**`LocationDraft` (`runebound-models/src/drafts.rs`) is unchanged.** The locked
answers (control archetype, resources, export mode, geography, site focus, base
protection/purpose, etc.) live **only in the wizard accumulator**; they are baked
into the generated prose and the derived/locked fields. We are *not* persisting
the raw answers, so no `*Draft`/`*Frontmatter`/`*Row` columns, no migration, no
`models.ts` regeneration for new fields.

> Consequence: there is **no** `cargo build -p runebound-models` step *for new
> fields*. The only model-crate edit is removing the `dungeon` kind (§6), which is
> Rust-internal (the enum is not `#[derive(TS)]`).

`exports` suppression uses an **empty `Vec<String>`**. Two small touch-ups make
empty read cleanly (see §5):

- `location_entity_card` (`drafts.rs`): skip the `Exports:` row when `exports` is
  empty (today it always renders).
- The wizard's generated draft sets `exports = vec![]` for Site/Hideout instead of
  the `["Unknown"]` default that `normalize_exports` produces.

## 2. The wizard (`desktop/src-tauri/src/wizards/location.rs`)

New file. Structure mirrors `wizards/dungeon.rs` exactly: accumulator + `*_data`/
`*_data_mut` downcast helpers, one struct per step, the `Wizard` impl, then
shared helpers. Build every prompt with `wizard::prompt::{wizard_menu,
choice_lines, action_row}` so choices are clickable by construction.

### 2.1 Accumulator

```rust
#[derive(Debug, Clone, Default)]
struct LocationWizardData {
    // Step 1
    kind_type: String,            // "town", "ruin", "hideout", "guildhall", "other"
    kind_custom: Option<String>,  // set when kind_type == "other"

    // Settlement (Q-A…Q-D)
    control: Option<String>,      // archetype label or resolved faction name
    faction_ref: Option<String>,  // faction slug/id when linked (read-only)
    resources: Option<String>,    // Q-B free text
    export_mode: Option<String>,  // "raw" | "refined" | "mixed"

    // Site (Q-S1…Q-S3)
    site_focus: Option<String>,   // "past" | "present" | "balanced"
    site_draw: Option<String>,    // optional override, else None → LLM picks

    // Hideout (Q-H1…Q-H4)
    base_owner: Option<String>,   // archetype label or resolved faction name
    base_protection: Option<String>, // "secrecy" | "force" | "both" | "barely"
    base_purpose: Option<String>, // "raids" | "smuggling" | "refuge" | "ritual" | "vault"

    // GM-locked danger for Site + Hideout (Q-S2 / Q-H3)
    danger_lock: Option<String>,  // "safe"|"guarded"|"risky"|"deadly"|"Unknown"

    // Shared optional map anchor (Q-D / Q-S4 / Q-H5)
    geography: Option<String>,

    // Custom / minimal seed (free text)
    custom_seed: Option<String>,

    // Generated seed held for the review screen, plus a one-shot notice.
    seed: Option<LocationSeed>,
    notice: Option<String>,
}
```

`LocationSeed` is reused from `services/ai_generation.rs` (it already has every
LLM-derived field). The accumulator is `Box<dyn Any>` to the engine
(`WizardData`); use the same `downcast_ref`/`downcast_mut` helper pattern dungeon
uses.

### 2.2 Step list & branching

`steps()` returns this **order** (order matters because intra-branch advance uses
`Next` = cursor+1; inter-branch jumps use `Goto`):

| # | Step `id()` | Branch | On accept |
| --- | --- | --- | --- |
| 1 | `kind` | router | `Goto` to the branch entry (below), or `Goto("custom_seed")` |
| 2 | `control` | Settlement (Q-A) | `Next` |
| 3 | `resources` | Settlement (Q-B) | `Next` |
| 4 | `export_mode` | Settlement (Q-C) | `Next` |
| 5 | `geography_settlement` | Settlement (Q-D, optional) | **generate** → `Goto("review")` |
| 6 | `site_focus` | Site (Q-S1) | `Next` |
| 7 | `site_danger` | Site (Q-S2, GM-lock) | `Next` |
| 8 | `site_draw` | Site (Q-S3, optional) | `Next` |
| 9 | `geography_site` | Site (Q-S4, optional) | **generate** → `Goto("review")` |
| 10 | `base_owner` | Hideout (Q-H1) | `Next` |
| 11 | `base_protection` | Hideout (Q-H2) | `Next` |
| 12 | `base_danger` | Hideout (Q-H3, GM-lock) | `Next` |
| 13 | `base_purpose` | Hideout (Q-H4) | `Next` |
| 14 | `geography_hideout` | Hideout (Q-H5, optional) | **generate** → `Goto("review")` |
| 15 | `custom_seed` | Minimal/custom | **generate** → `Goto("review")` |
| 16 | `review` | shared terminal | `continue`→`Complete`; `reroll`→regenerate+`Stay`; `cancel` |
| (17) | `faction_link` | shared (increment 7) | sets `control`/`base_owner`, `Goto` back |

**Step 1 (`kind`) routing** — `accept()` normalises the chosen kind and returns:

```rust
match kind {
    "hamlet" | "town" | "city"      => Goto("control"),
    "ruin"  | "landmark" | "wilderness" => Goto("site_focus"),
    "hideout"                        => Goto("base_owner"),
    "guildhall"                      => Goto("custom_seed"), // minimal branch
    "other" (option 0 / free text)   => Goto("custom_seed"), // captures kind_custom first
    _ => Stay,
}
```

The kind step presents the nine kinds as a numbered `wizard_menu` (`dungeon` is
absent — see §6) plus **option 0: custom**. Picking `0` should accept follow-up
free text as `kind_custom` (set `kind_type="other"`). Simplest implementation:
option `0` echoes "type a custom kind name", and the next submission on the same
step sets `kind_custom` + `kind_type="other"` and `Goto("custom_seed")`. (A
dedicated `custom_kind` micro-step is an acceptable alternative; keep it inline to
avoid step sprawl.)

`guildhall` → `custom_seed` with `kind_type="guildhall"` (design §4d: "minimal →
straight to generate"; it's owned-but-public, no subpath).

**Why each branch's last step generates:** `geography_*` (and `custom_seed`) are
always the final, optional step of their branch. Their `accept()` records the
(possibly skipped) geography, calls the shared `generate_location_into(data,
state)`, and `Goto("review")`. Set `awaiting_llm_label() = Some("generating
location")` on **steps 5, 9, 14, 15** so the frontend spinner fires on submission
(the structured `WizardView` carries it — no `commandSpinnerLabel` hint needed for
wizard steps; see §8).

**Engine note:** `Goto`/`Next` push history, so `back` works at every step
(`runtime.rs::run_transition`). No per-wizard nav code.

### 2.3 Per-step prompts & choices

Use the design §4 question text verbatim for prompts. Choice tokens are short
(`1`,`2`,…, or the option keyword). Examples:

- `control` (Q-A): `1 noble house / lord`, `2 faction or guild`, `3 council / free
  city`, `4 independent / contested`, `5 let the model decide`, plus (increment 7)
  `link` → `Goto("faction_link")`.
- `export_mode` (Q-C): `1 raw`, `2 refined`, `3 mixed`.
- `site_focus` (Q-S1): `1 what it was`, `2 what's here now`, `3 balanced`.
- `site_danger` / `base_danger` (Q-S2/Q-H3): `1 safe`, `2 guarded`, `3 risky`,
  `4 deadly`, `5 let the model decide` (→ `danger_lock = "Unknown"`). Validate
  against `LOCATION_DANGER_LEVELS`.
- `base_protection` (Q-H2): `1 secrecy`, `2 force`, `3 both`, `4 barely`.
- `base_purpose` (Q-H4): `1 raids`, `2 smuggling`, `3 refuge`, `4 ritual`,
  `5 vault`.
- `base_owner` (Q-H1): `1 faction / guild`, `2 a single operator`, `3 a creature /
  monster`, `4 a cult`, plus (increment 7) `link`.
- Optional steps (`geography_*`, `site_draw`) offer a clickable `skip`
  (token `skip`) and accept free text otherwise — copy `ContextStep` in
  `dungeon.rs` verbatim for the skip/free-text shape.

Each step gets a one-line `summary()` for in-wizard `help` (engine renders it).

### 2.4 `Wizard` impl

```rust
impl Wizard<AppState> for LocationWizard {
    fn id(&self) -> &'static str { "location" }
    fn title(&self) -> &'static str { "Create Location" }
    fn steps(&self) -> &[Arc<dyn WizardStep<AppState>>] { &self.steps }
    async fn seed(&self, _host) -> Result<WizardData, String> {
        Ok(WizardData::new(LocationWizardData::default()))
    }
    async fn finalize(&self, state, d) -> CommandResult { /* see 2.5 */ }
}
```

### 2.5 `finalize()` — hand off to the location editor

No LLM. Build a `LocationDraftSession` (= `LocationDraft`) from the accumulator's
generated `seed` + locked answers, then open the editable draft exactly as the
current one-shot create does (`create_commands.rs::create_location` lines
166–191 are the template):

```rust
let seed = data.seed.clone().ok_or("location flow reset; run create location again")?;
let draft = LocationDraftSession {
    id: make_entity_id("loc"),
    seed_prompt: build_seed_prompt(data),   // persist locked answers as bias for later reroll
    slug: slugify(&seed.name),
    name: seed.name,
    vault_path: String::new(),
    kind_type: data.kind_type.clone(),       // GM-locked from step 1 (NOT the model's)
    kind_custom: data.kind_custom.clone(),
    visual_description: seed.visual_description,
    history_background: seed.history_background,
    exports: seed.exports,                   // derived (settlement) or [] (site/hideout)
    tone: seed.tone,
    authority: seed.authority,               // shaped by Q-A/Q-H1 (or faction name)
    danger_level: seed.danger_level,         // GM-locked for site/hideout, LLM for settlement
    current_tension: seed.current_tension,
};
{ let mut editor = state.editor_session.lock().await; editor.set_location(draft.clone()); }
command_response_with_event(
    prepend_notice(data.notice.clone(), location_summary_text(&draft)),
    location_event_from_draft(&draft),
)
```

Reuse `location_summary_text` / `location_event_from_draft` from
`entities/domains/location_domain.rs` (already `pub`). `kind_type`/`kind_custom`
come from the **accumulator**, not the seed — that is the whole point: kind is
GM-locked at step 1.

`build_seed_prompt(data)` flattens the locked answers into a single bias string
(mirror dungeon's `build_seed_prompt`) so a later `location reroll <field>` in the
entity editor reuses the GM's intent (the reroll service already merges
`seed_prompt`, see `location_domain.rs::reroll_field` + `merge_seed_and_reroll_prompt`).

### 2.6 `review` step

Mirror `StoryReviewStep` in `dungeon.rs`:

- `prompt`: render `location_entity_card(&draft_preview, CardFooter::Hide)`
  built from the accumulator seed + locked answers (or a lightweight summary), then
  an `action_row` of `continue · reroll · cancel`. Show `data.notice` once if set.
- `choices`: `continue` (→ build the location), `reroll` (→ regenerate, optional
  `reroll <hint>`), `cancel`.
- `accept`: `continue`→`Complete`; `reroll [hint]`→`generate_location_into(data,
  state, hint)` + `Stay`; else `Stay`.
- `awaiting_llm_label() = Some("generating location")`.

> **Frontend wiring for review `reroll`:** `wizardSpinnerLabel` (App.tsx) treats
> `reroll` as a *local* action (no spinner) **except** the dungeon `story_review`
> special-case. Add the sibling line for location:
> `if (wizard.step_id === "review" && isReroll) return "generating location";`
> (see §8). Alternatively name the regenerate choice `regenerate` (not in the
> local-action list, so the label fires automatically) — but `reroll` matches the
> dungeon UX, so prefer the one-line special-case.

## 3. Kind-aware generation (`services/ai_generation.rs`)

Add one method alongside `generate_location_seed`:

```rust
pub async fn generate_location_seed_for_wizard(
    &self,
    inputs: &LocationWizardInputs,   // a plain struct of the locked answers (defined in ai_generation.rs)
    database: &Database,
    generation_repo: &dyn GenerationRepository,
) -> Result<SeedGeneration<LocationSeed>, String>
```

It reuses the existing scaffolding — `load_generation_config`,
`build_reference_context`, `run_seed_attempts`, `LOCATION_GEN_SAMPLING`,
`normalize_location_seed`/`validate_location_details` — and differs only in:

1. **Schema shape per kind** (the `serde_json::json!` schema passed to
   `run_seed_attempts`):
   - **Settlement:** today's schema (keep `exports`, `authority`, `danger_level`).
   - **Site / Hideout:** drop `exports` from `required` + `properties` and drop
     `danger_level` (GM-locked, injected after). Keep `authority` (occupant/owner)
     and all prose fields.
   - **Custom/guildhall:** this method is **not** used — the minimal/custom branch
     calls the existing `generate_location_seed(custom_seed)` unchanged (model's
     call for everything, per §5 Custom column).
2. **System/user prompt** embeds the locked answers as authoritative context:
   - Settlement: "This is a {kind}. It is controlled by {control}. Natural
     resources: {resources}. Exports are **{export_mode}** — produce a 1–3 item
     `exports` list consistent with that (raw → ship the resource as-is; refined →
     the processed/finished good; mixed → both). {geography?}." → this is how
     `exports` becomes *derived* not invented.
   - Site: "This is a {kind}, weighted toward {site_focus}. {draw?} {geography?}.
     Do not invent rulers or exports." Danger is GM-set; write the *source* of the
     danger in `current_tension`/`history` but do not contradict the level.
   - Hideout: "This is a base owned by {base_owner}, protected by
     {base_protection}, used for {base_purpose}. {geography?}. Present-tense; no
     exports." `authority` = the owner.
3. **Post-processing after `run_seed_attempts` returns** (or inside the `accept`
   closure):
   - `kind_type`/`kind_custom`: **ignore** the model's; overwrite from the locked
     accumulator. (Equivalently, don't put `kind_type` in the wizard schema at
     all.)
   - Site/Hideout: `seed.exports = vec![]` (suppressed); `seed.danger_level =
     inputs.danger_lock` (GM value, default `"Unknown"` for "let the model
     decide").
   - Settlement with a faction link: set `seed.authority` to the linked faction's
     canonical name (increment 7).

> Keep `normalize_location_seed` honest: it currently requires `exports` 1–3 and
> `kind_custom` rules. For Site/Hideout we deliberately set `exports = []` *after*
> normalisation, or relax the wizard path to skip the `exports` length check when
> suppressed. Simplest: run `validate_location_details` only on the
> settlement/custom path; for site/hideout validate the prose fields but skip the
> exports/danger checks (those are GM-owned/suppressed). Factor the shared field
> validation so both paths reuse it.

`LocationWizardInputs` is a borrow-friendly struct the wizard fills from
`LocationWizardData` and passes in (keeps `ai_generation.rs` free of wizard
types). The wizard's `generate_location_into` helper:

```rust
async fn generate_location_into(d: &mut LocationWizardData, state: &AppState, hint: Option<&str>) -> Result<(),String> {
    let ai = AiGenerationService;
    let (db, repo) = (state.database(), state.generation_repo());
    let gen = if is_structured(d.kind_type) {
        ai.generate_location_seed_for_wizard(&d.as_inputs(hint), db.as_ref(), repo.as_ref()).await?
    } else {
        ai.generate_location_seed(d.custom_seed.clone(), db.as_ref(), repo.as_ref()).await? // minimal/custom
    };
    d.seed = Some(gen.seed);
    d.notice = gen.notice;
    Ok(())
}
```

## 4. Command surface (`command-specs` + `create_commands.rs`)

### 4.1 Launch routing — `commands/create_commands.rs::handle_create`

Replace the location arm. **Bare** `create location` launches the wizard; the
`<prompt>` form keeps the one-shot handler:

```rust
if lowered == "create location" {
    return crate::wizards::start_wizard("location", invocation.state.inner()).await;
}
if lowered.starts_with("create location ") {
    return create_location(trimmed, invocation.state.clone()).await; // existing one-shot, unchanged
}
```

(Mirror the dungeon arm directly above it. Keep `create_location` exactly as-is —
it is now the "custom lane" the design preserves.)

### 4.2 `command-specs/src/lib.rs`

- **No new command, no new `command_availability` arm.** `create` exists; the
  wizard nav verbs (`continue`/`back` = `AnyWizard`, `cancel` =
  `AnyEditorOrWizard`) already have arms; the kind-picker choices are *wizard
  choices*, not manifest commands.
- Update the `create` spec's `location` subcommand summary to note it's guided
  (e.g. "Start the guided location flow (or `create location <prompt>` for
  one-shot)"), matching the dungeon subcommand's phrasing. Examples already cover
  both forms — leave them.
- **Spinner hints** (`spinner_hints`, ~lib.rs:240): **keep**
  `("create location", "generating location")`. It serves the one-shot
  `create location <prompt>` path. Bare `create location` launches the wizard and
  returns its first prompt instantly; the optimistic spinner is replaced on the
  same render. When a wizard is *active*, `commandSpinnerLabel` returns early via
  `wizardSpinnerLabel` (App.tsx:923), so the hint never double-fires inside the
  flow. (Dungeon omits its hint because it has *no* one-shot form; location keeps
  one because it does.)

## 5. Conditional fields & card (`runebound-models/src/drafts.rs`)

Per design §5, suppression is enforced at **generation** (the model never emits a
suppressed field). The card only needs to render an empty list gracefully:

- `location_entity_card`: push the `Exports:` row **only when** `draft.exports`
  is non-empty (after `normalize_unknown_list`). Today it always renders; for
  Site/Hideout (`exports == []`) the row should be omitted, not shown as
  "Unknown". One conditional around the `entity_row("Exports:", …)` push.
- Everything else (`authority`, `danger_level`, prose) always renders; their
  *values* differ by kind but the rows are universal (design §5 table).

No other card or field-visibility change — `location set`/`reroll` over the full
field set stays valid (a GM can still hand-edit a suppressed field afterward;
that's fine and matches "model's call" leniency).

## 6. Remove the `dungeon` location kind (build order step 1)

This is the **standalone `Dungeon` entity's** kind list of *locations*, not the
`Dungeon` entity itself. The entity, its table, `EntityKind::Dungeon`, the
`create dungeon` wizard, `vault_sync` `KIND="dungeon"`, `entity_admin`'s
`"dungeon"` arm, and `publish`'s `doc_type:"dungeon"` are **all untouched** — they
refer to the Dungeon entity, which the design keeps (§2 "Dungeons: Unchanged").

Edits (search target: the location kind constant/enum):

1. `runebound-models/src/utils.rs`:
   - `LOCATION_KIND_TYPES`: drop `"dungeon"` → `pub const LOCATION_KIND_TYPES: [&str; 9]`.
   - `enum LocationKindType`: remove the `Dungeon` variant and its `as_str` arm.
     (The enum is not `#[derive(TS)]` and `LocationDraft.kind_type` is a `String`,
     so this is Rust-internal — no `models.ts` change.) Grep confirms `Dungeon` is
     referenced only here.
2. `desktop/src-tauri/src/entities/domains/location_domain.rs`:
   `normalize_location_kind_type` has a **duplicated** local `LOCATION_KIND_TYPES`
   (lines 323–334). Drop `"dungeon"` → `[&str; 9]`. **Recommended cleanup:** delete
   the local copy and import `runebound_models::utils::LOCATION_KIND_TYPES` so the
   list lives in one place (the duplication is pre-existing tech debt this change
   should not extend).
3. `services/ai_generation.rs` already reads `LOCATION_KIND_TYPES` from
   `runebound_models::utils` (the generation schema enum follows automatically).
4. **DB:** none. `kind_type` is a `lenient … = "other"` TEXT column
   (`impl_entity_table!` in `core/src/db.rs`), no enum constraint, so old rows
   never block; design accepts purged test vaults (§2), so no migration.

Acceptance: `create location` no longer offers `dungeon`; `location set kind_type
dungeon` is rejected with the 9-kind list; `cargo test --workspace` green
(including the desktop registry/schema contract tests).

## 7. Read-only faction link (build order step 7, last)

Adds an optional `faction_link` step reachable from `control` (Q-A) and
`base_owner` (Q-H1) via a `link` choice (`Goto("faction_link")`):

- `prompt`/`choices`: list existing factions as clickable choices. Source:
  `state.faction_repo().list_all(db.as_ref())` (or `search_by_name` if the GM
  types a filter) — both already on `FactionRepository`
  (`repositories/mod.rs`). **Reads only; never `upsert`/writes.**
- `accept`: a chosen faction sets `control`/`base_owner` to that faction's
  canonical `name` and records `faction_ref = slug`; then `Goto` back to the
  branch's next step (resources for settlement; protection for hideout). Offer a
  `skip` that returns without linking.
- Generation: when `faction_ref` is set, post-processing forces `seed.authority`
  to the faction name (so the ruler is the known house, not an invented one —
  design §4a/§4c).

This ships in 0.6.0 but is the final increment; gate it behind the structured
branches landing first. No `command_specs` change (it's a wizard step).

## 8. Frontend (`desktop/src/App.tsx`) — minimal

The wizard drives its own spinner through the structured `WizardView`
(`awaiting_llm_label`); `finalize()` emits `LoadLocationDraftWithCard` (via
`location_event_from_draft`), which App.tsx already handles for one-shot create.
