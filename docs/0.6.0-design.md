# 0.6.0 Design Note — The Location Wizard

> **Status:** Draft / discussion. Scope is locked to **locations only** for 0.6.0.
> **Theme:** A location should know what it is *before* the model writes a word about it. The GM answers the questions only a map can answer; the model fills in the prose under those answers.

---

## 1. The Core Insight

The schema already supports far more variety than the creation flow exposes. `LocationRow.kind_type` (`core/src/db.rs`) already enumerates `hamlet, town, city, hideout, ruin, guildhall, landmark, wilderness, other`, and locations already carry `tone`, `danger_level`, `visual_description`, `history_background`, `current_tension`.

The friction is that today's one-shot generation lets the model **invent things the GM already knows**:

- It hallucinates `exports` from nothing, when a GM with a map knows exactly what the land produces and how it ships.
- It defaults `authority` to a guild or collective, when a real town is usually run by a single person or house.
- It treats every location like a settlement, even a ruin that exports nothing and has no ruler.

### The reframe: GM-locked vs LLM-derived

Convert location creation from one-shot to a **wizard** that partitions every field into two classes:

- **GM-locked** — answered by the GM up front. These are facts about the world the model has no business inventing. *The exact locked set varies by kind* (see §4): a settlement locks control + resources + export mode; a site locks its danger.
- **LLM-derived** — generated *under* the locked answers: name, visual description, history, tone, the prose fields, and any list (like `exports`) computed from locked inputs.

So `exports` stops being invented and becomes **derived** from the GM's inputs. And because `kind_type` is chosen at step 1 — a stable, non-LLM value — it becomes the routing key that decides which questions are asked and which fields even exist for this location.

## 2. Decisions Locked

| Decision | Resolution |
| --- | --- |
| Creation model | One-shot → **wizard** |
| Step 1 | **Kind picker** (current kinds, plus a custom/freeform option) |
| `dungeon` as a location kind | **Removed entirely.** Breaking change accepted — no users, test vaults are purged on deploy, so no migration needed |
| Fast path | `create location` → wizard; `create location <prompt>` → drops into the **custom branch** pre-filled and generates immediately. The one-liner survives as the custom lane |
| Settlement control (Q-A) | Widened beyond house/faction (see §4a) so merchant republics aren't forced into "house" |
| Export mode (Q-C) | **Three-way:** raw · refined · mixed |
| Geography (Q-D / Q-S4 / Q-H5) | **Optional, skippable** prompt that grounds output to the GM's map |
| `danger_level` ownership | **LLM-derived for settlements; GM-locked for sites & hideouts** — for a town danger is incidental, for a site it's the point |
| `guildhall` routing | **Its own faction-locked branch (§4e).** Now that the read-only faction link has landed, a guildhall opens straight on a *mandatory* faction picker (an unknown name is accepted as a new ad hoc faction, so the GM is never blocked); it then asks the hall's public role and **which existing location it stands within** (a second picker over the GM's locations, or a typed new place). `authority` locks to the faction, `exports` are suppressed, and `danger_level` is LLM-derived (a public hall's danger is incidental, like a settlement's) |
| Read-only faction link (Q-A / Q-H1) | **Ships in 0.6.0.** The wizard can point control/ownership at an existing faction entity. Reads factions, never writes them; skippable |
| Factions (otherwise) | **Out of scope** for 0.6.0 (deferred) — the read-only link above is the one exception |
| Dungeons | **Unchanged.** The standalone Dungeon entity stays as-is; removing the location *kind* `dungeon` simply ends the dual representation. Location↔dungeon coupling revisited in 0.7 |

## 3. Architectural Rule (keeps the blast radius small)

The wizard is **not a parallel entity**. It produces the *exact same* `LocationDraft` (`runebound-models/src/drafts.rs`) as one-shot generation ever did.

- The custom branch (and `create location <prompt>`) is just the old one-shot path, reachable from inside the wizard.
- Every branch converges on one `LocationDraft`, so `save` / `reroll` / field-edit / the card UI all work unchanged.

This mirrors how `create dungeon` already feeds `DungeonDraft` via `desktop/src-tauri/src/wizards/dungeon.rs` — we reuse that scaffolding, not reinvent it.

## 4. The Wizard, Step by Step

### Step 1 — Kind (routing key)

Pick one of the existing kinds, or **option 0: custom** (free text → sets `kind_custom`). `dungeon` is gone from this list. The choice routes to one of four branches:

| Kind | Branch |
| --- | --- |
| `hamlet` · `town` · `city` | **Settlement** (§4a) |
| `ruin` · `landmark` · `wilderness` | **Site** (§4b) |
| `hideout` | **Hideout / lair / base** (§4c) |
| `guildhall` | **Guildhall** (§4e) — a faction's public HQ |
| custom / freeform | **Minimal** → straight to generate (§4d) |

### 4a. Settlement branch

A real town exists for a reason a GM already knows. These questions capture that reason instead of letting the model guess it.

- **Q-A — Who controls it?** `noble house / lord` · `faction or guild` · `council / free city` · `independent / contested` · `let the model decide`
  - *Why:* the model over-picks guilds and collectives; most towns answer to one person or house. The `council / free city` option exists so genuine merchant republics aren't mislabeled.
  - *Faction link (ships 0.6.0, skippable):* point Q-A at an **existing faction entity** (factions already exist in the app — the wizard only reads them, never writes them) so the ruler is a known house rather than an invented one.
- **Q-B — What natural resources are here?** Free text — the GM's map knowledge. Towns serve a purpose; they don't exist out of thin air.
- **Q-C — Are its exports raw, refined, or mixed?** `raw` · `refined` · `mixed`
  - *Why:* transport logistics. A farming town on a river floats raw grain downstream. A town three days from any river, rail, or sea can't move bulk grain, so it must refine into shippable goods. The GM has already done this reasoning from the map; Q-C just records the conclusion.
- **Q-D — Geography / trade route?** *(optional, skippable)* e.g. "on a river," "landlocked, four days from the coast." Enriches the narrative fields. Q-C carries the logistics; this adds color.

`exports` is then **derived** from Q-B + Q-C, not invented.

### 4b. Site branch — `ruin` · `landmark` · `wilderness`

A non-settlement you stumble on. No exports, no ruler, so those fields are suppressed. The kind hints whether the place is about its past or its present; the wizard makes that explicit and lets the GM own the one thing that's the point of a site — its danger.

- **Q-S1 — Is this place about what it *was*, or what's *here now*?** `what it was` · `what's here now` · `balanced`
  - *Why:* weights generation toward `history_background` vs. occupant + `current_tension`. The kind hints it, but a sacred grove and a haunted grove diverge — let the GM pick.
- **Q-S2 — Danger level.** GM picks `safe` · `guarded` · `risky` · `deadly` (or *let the model decide* → `Unknown`). The **LLM generates the source and how it manifests.**
  - *Why the inversion from settlements:* for a found site, danger is the entire point, so the GM dials the level and the model writes the *why*.
- **Q-S3 — The draw.** *(optional)* Defaults to the LLM; the GM can override with a reason players come here: loot · quest objective · passage / shortcut · a person who lives here · buried lore.
  - *Why optional:* a site needs a reason to exist, but the GM is usually happy to let the model surprise them with it.
- **Q-S4 — Geography / map anchor.** *(optional)* "Half a day from Greenhollow, deep in the marsh." Grounds output to the GM's map.

### 4c. Hideout / lair / base branch — `hideout`

Someone's base — **owned**, **deliberately protected**, and **present-tense by definition** (an abandoned base is just a ruin, so there's no past/present pivot here). `authority` is repurposed as the owner; `exports` stay suppressed.

- **Q-H1 — Whose base is it?** `faction / guild` · `a single operator` (chief, hermit, rogue mage) · `a creature / monster` · `a cult`
  - *Faction link (ships 0.6.0, skippable):* link an **existing faction entity** — read-only, same as settlement Q-A.
- **Q-H2 — Protected by what?** `secrecy` · `force` · `both` · `barely`
  - *Why one axis:* this single question covers both "how hidden" and "how fortified." A smuggler's cache is secrecy-only; a warlord's keep is force. It drives how players find the base and what defenses the model writes.
- **Q-H3 — Danger level.** Same as Q-S2: GM picks the level, the LLM writes the source.
- **Q-H4 — Purpose.** What the base is *for*: `raids` · `smuggling` · `refuge` · `ritual` · `vault`.
  - *Why asked outright (not deferred like Q-S3):* a base is intentional — the GM placed those bandits because they raid the road. The purpose is plot, not flavor.
- **Q-H5 — Geography / map anchor.** *(optional)* Where the base hides — the city's sewers, a mountain pass, deep woods.

### 4d. Minimal / custom branch

Custom kinds and freeform prompts (`create location <prompt>`) skip the structured questions and go straight to generation with the user's text as the seed. This is the preserved one-shot fast path.

### 4e. Guildhall branch — `guildhall`

A guildhall is, by definition, a faction's **public headquarters** — without a faction it's just a generic building, and the model would be inventing the one fact the GM most wants to own. So the guildhall branch is **faction-locked**: it opens directly on the shared faction picker (the same read-only typeahead settlements/hideouts use), and the link is **mandatory**.

- **Q-G1 — Which faction runs this hall?** The faction picker, but **required** (no skip). Matching an existing faction (DB draft or published note) links it; **typing a name that matches nothing is accepted as a brand-new faction reference**, so a GM with no factions yet is never blocked.
  - *Rich context:* a matched **published** faction's authoritative metadata (agenda, methods, leadership, goals, tone) flows into generation via the existing `@reference` machinery — emitted as `@factions/<Name>` in the seed prompt — so a Crimson Lanterns hall and a Silver Hand hall come out genuinely different. Draft and free-typed names contribute their name only (the `@reference` index covers published notes); this is the same trade-off the rest of the app's cross-entity context makes, and could be deepened later.
- **Q-G2 — What is the hall's public role?** *(optional)* `counting house / bank` · `training hall` · `trade exchange` · `lodge / chapterhouse` · `courthouse / tribunal` — pick one, type your own, or skip (the model decides). Lets two halls of the same faction differ.
- **Q-G3 — Where does it stand?** **Required** — a hall stands somewhere. A second picker (same typeahead, this time over the GM's **existing locations**): pick one, or **type a new place name** if the containing location isn't in the world yet (so the GM is never blocked). A matched **published** location flows in as authoritative context via `@locations/<Name>`, the same `@reference` machinery the faction link uses.

`authority` locks to the faction; `exports` are suppressed (a hall isn't a settlement economy); `danger_level` stays LLM-derived (public, incidental).

### Generate → Review

Generation fills the LLM-derived fields under the locked answers; review hands off to the normal editor.

## 5. Conditional Fields by Kind

Because kind is known *before* generation, field visibility and the generation schema are decided up front — the model never invents a field that doesn't apply.

| Field | Settlement | Site | Hideout | Guildhall | Custom |
| --- | --- | --- | --- | --- | --- |
| `exports` | shown, derived from Q-B/Q-C | suppressed | suppressed | suppressed | model's call |
| `authority` | shown, shaped by Q-A | occupant / guardian (or none) | the base owner (Q-H1) | **the linked faction (locked, Q-G1)** | model's call |
| `danger_level` | LLM-derived | **GM-locked** (Q-S2) | **GM-locked** (Q-H3) | LLM-derived | model's call |
| `tone`, `visual_description`, `history_background`, `current_tension` | shown | shown | shown | shown | shown |

## 6. Build Order

1. **Step-1 kind picker + routing** — the spine; remove `dungeon` from the kind list/enum.
2. **Settlement branch (Q-A…Q-D)** — wire Q-B/Q-C into derived `exports` and Q-A into `authority`.
3. **Conditional fields + kind-aware generation prompt** (`desktop/src-tauri/src/services/ai_generation.rs`) — suppress/derive/GM-lock per §5.
4. **Custom branch + `create location <prompt>` fast path.**
5. **Site subpath (Q-S1…Q-S4).**
6. **Hideout subpath (Q-H1…Q-H5).**
7. **Read-only faction link (Q-A / Q-H1)** — a picker over existing faction entities that sets control/ownership; reads, never writes.
8. **Guildhall branch (§4e)** — promote `guildhall` from the minimal lane to its own faction-locked branch: route it into the faction picker in a mandatory mode (no skip; an unknown name becomes a new ad hoc faction), reuse the picker wholesale (extract a shared entity typeahead/match helper), add a public-role step and a **location-anchor** step (a second mandatory picker over existing locations, or a typed new place), and add a `Guildhall` generation branch (exports suppressed, `authority` locked to the faction, `danger_level` LLM-derived, `@factions/<name>` + `@locations/<name>` carried into the prompt for rich context).

Wizard plumbing follows the existing pattern: command metadata in `command-specs/src/lib.rs`, wizard logic alongside `desktop/src-tauri/src/wizards/dungeon.rs`, draft contract in `runebound-models`.
