# Kind-based subfoldering for published location notes

## Context

Today every location publishes flat to `locations/<Name>.md`, regardless of kind
(settlement, site, hideout, guildhall, other). As campaigns grow, the folder
becomes an undifferentiated pile. We want published location `.md` files sorted
into kind-based subfolders so the Obsidian vault stays navigable. This is purely
an **Obsidian vault layout** change — the canonical TOML store and the SQLite
projection stay flat and untouched.

The subtlety is the `@reference` system, which is **path-keyed**: a note at
`locations/settlements/Foo.md` is referenced as `@locations/settlements/Foo`, not
`@locations/Foo`. The guildhall wizard emits a `@locations/<anchor>` seed to
ground generation in the containing place, so subfoldering must thread the
subfolder through that one seed or the grounding silently degrades to name-only.
Obsidian `[[wikilinks]]` resolve by note name regardless of folder, so the
published `Authority:`/`Location:` links need no change.

### Locked decisions (from the user)
- **Taxonomy = by branch (4 folders)**, reusing the existing `location_branch()`
  grouping: `locations/settlements/` (hamlet·town·city), `locations/sites/`
  (ruin·landmark·wilderness), `locations/hideouts/`, `locations/guildhalls/`.
  `other`/freeform/unknown kinds stay **flat** in `locations/`.
- **One-shots always flat.** Only the multi-step wizard subfolders. The quick
  one-shot lane (`/create location <prompt>`) carries an LLM-chosen kind with no
  lane marker, so kind alone can't distinguish it — a creation-lane signal is
  required.
- **No migration.** Existing flat `.md` files stay put; we add no backfill and do
  not wire `move_file`. Orphan-on-republish (a renamed/refolded note leaves its
  old `.md` behind) is accepted.

## Design

### 1. Subfolder helper (`desktop/src-tauri/src/services/ai_generation.rs`, near `location_branch` ~L1330)
`location_branch()` can't express "other → flat" (its `_ => Settlement` fallback
would route `other` into `settlements/`), so match kind strings directly:

```rust
/// Under-`locations/` subfolder for a structured wizard kind, or None for
/// other/freeform/unknown (stays flat). Only shapes the readable .md vault_path.
pub fn location_subfolder(kind_type: &str) -> Option<&'static str> {
    match kind_type {
        "hamlet" | "town" | "city"          => Some("settlements"),
        "ruin" | "landmark" | "wilderness"  => Some("sites"),
        "hideout"                           => Some("hideouts"),
        "guildhall"                         => Some("guildhalls"),
        _ => None, // other / freeform / unknown -> flat
    }
}

/// Full relative dir for a NEW location save: `locations/<sub>` or flat `locations`.
pub fn location_dir_for_kind(base: &str, kind_type: &str) -> String {
    match location_subfolder(kind_type) {
        Some(sub) => format!("{base}/{sub}"),
        None => base.to_string(),
    }
}
```

### 2. Creation-lane signal — transient field on the draft (`runebound-models/src/drafts.rs`, `LocationDraft`)
The canonical store persists `LocationFrontmatter` (flat, slug-keyed) and the DB
row is `LocationRow`; neither is the draft. `LocationDraft` (= `LocationDraftSession`)
is the in-memory session type and **is** ts-exported. Add a transient flag, kept
out of TOML, DB, and `models.ts`:

```rust
/// Transient: true only when the WIZARD built this draft, requesting kind-based
/// subfoldering of the .md vault path. Never persisted, never sent to the frontend.
/// Consulted ONLY when persisting a brand-new row; existing rows preserve their
/// on-disk folder regardless (so this defaulting to false on load is harmless).
#[serde(skip)]
#[ts(skip)]
pub wizard_subfoldered: bool,
```

`bool: Default`, so `#[serde(skip)]` reads as `false` on every store/DB load. This
survives wizard→save because `save_active_draft` saves the editor-session draft
object directly (no DB round-trip). Set it in `build_location_draft`
(`wizards/location.rs` ~L1509) → `wizard_subfoldered: true`. Every other struct
literal must add `wizard_subfoldered: false` (compiler enumerates them via
struct-completeness errors): `create_commands.rs` one-shot (~L172),
`location_domain.rs` `resolve` (~L69) and `normalized_draft` (~L352), `app_state.rs`
test helper, and test literals.

### 3. Path computation (`entity_persistence.rs` + `entity_persistence_macros.rs`)
The macro (shared by 7 entities) calls `resolve_vault_path(.., $dir, name, existing)`
with the `$dir` literal at macros L75; `normalize` (L56) has already bound
`kind_type`. Two changes, both no-ops for the other 6 entities:

**a. Optional macro arg** (`entity_persistence_macros.rs`): add after `dir: $dir:literal,`
```
$( vault_dir: $vault_dir:expr, )?
```
Just before the `resolve_vault_path` call, compute the dir (expr may read `draft`
and `kind_type`, same hygiene as `frontmatter_fields`):
```rust
let vault_dir = { #[allow(unused_mut)] let mut d = $dir.to_string(); $( d = $vault_dir; )? d };
```
Pass `&vault_dir` instead of `$dir` to `resolve_vault_path`. Keep `resolve_slug`
on `$dir` ("locations") — slugs and the TOML store stay flat.

**b. Location invocation** (`entity_persistence.rs` ~L132): add
```rust
vault_dir: crate::services::ai_generation::location_dir_for_kind(
    "locations",
    if draft.wizard_subfoldered { &kind_type } else { "other" }, // "other" => flat for one-shots
),
```

**c. `resolve_vault_path` — preserve the folder for existing rows** (`entity_persistence.rs` ~L509).
Single dir param; new rows use it, existing rows derive their dir from the current
path's parent (so a re-save never thrashes a subfoldered note back to flat, nor a
flat one-shot into a subfolder):
```rust
async fn resolve_vault_path(document_repo, database, dir: &str, name, existing)
  -> Result<(String, Option<String>), String> {
    match existing {
        Some(cur) => {
            let dir = parent_dir_of(cur.vault_path).unwrap_or(dir); // preserve current folder
            let readable = unique_readable_vault_path(document_repo, database, dir, name, Some(cur.slug)).await?;
            if normalize_relative_path_for_storage(cur.vault_path)
                == normalize_relative_path_for_storage(&readable) {
                Ok((cur.vault_path.to_string(), None))
            } else { Ok((readable, Some(cur.vault_path.to_string()))) }
        }
        None => Ok((unique_readable_vault_path(document_repo, database, dir, name, None).await?, None)),
    }
}

/// `locations/sites/Foo.md` -> Some("locations/sites"); `Foo.md` -> None.
fn parent_dir_of(vault_path: &str) -> Option<&str> {
    let p = vault_path.trim_end_matches('/');
    p.rfind('/').map(|i| &p[..i])
}
```
For the 6 untouched entities, existing files are always directly under `$dir`, so
`parent_dir_of` returns `$dir` — identical behavior.

### 4. Anchor typeahead carries the subfolder (`wizards/location.rs`)
`entity_suggestions`/`match_entity` (~L1006/L1030) take `&[(String,String)]` and are
**shared with the faction picker**, so do not widen that tuple. Carry the subfolder
in a parallel location-only structure.

- **`entries_from_refs`** (~L1431): accept exactly one level of nesting and return
  the subfolder. New return `Vec<(String,String,String)>` = `(name, slug, sub)`
  where `sub` is `""` for `locations/Foo.md` and `"settlements"` for
  `locations/settlements/Foo.md`; reject deeper (`a/b/Foo.md`). Replace the
  `if file.contains('/') { continue; }` reject with a one-level split.
- **Faction path stays flat**: `load_published_entity_names("factions")` maps the
  3-tuple down with `.map(|(n,s,_)| (n,s))` (factions never nest, so `sub` is
  always `""`).
- **Location reader**: add `load_published_locations()` returning the subfolder.
- **`LocationAnchorChoice`**: small struct `{ name, slug, sub }` for the anchor list.
  `load_linkable_locations` (~L1378) returns `Vec<LocationAnchorChoice>`: DB drafts
  derive `sub` from their `kind_type` via `location_subfolder().unwrap_or("")`
  (a draft will publish to that subfolder); published notes take `sub` from the
  path. Add a location-specific merge (keep shared `merge_linkable` for factions).
- **Accumulator** (`LocationWizardData` ~L94): change `locations: Vec<(String,String)>`
  → `Vec<LocationAnchorChoice>`; add `location_anchor_sub: Option<String>` (~L73).
- **`GuildhallAnchorStep`** (~L871/886): `suggest`/`accept` project the choices to
  `(name, slug)` for the shared helpers; on `EntityMatch::Found` look the chosen
  slug back up in `locations` to set `location_anchor_sub = Some(choice.sub)`;
  on `EntityMatch::None` (free-typed) leave it `None`.

### 5. Reference emission (`ai_generation.rs`)
- Add `pub location_anchor_sub: Option<String>` to `LocationWizardInputs` (~L1343);
  set it in `as_inputs` (`wizards/location.rs` ~L131).
- `build_wizard_user_prompt` Guildhall arm (~L1568): emit
  `@locations/<sub>/<anchor>` when `sub` is non-empty, else `@locations/<anchor>`
  (free-typed fallback). `build_seed_prompt` inherits this via `as_inputs`.
- `draft.location` stays the **bare name** (Obsidian `[[name]]` resolves by name) —
  `build_location_draft` L1525 unchanged.

### Unchanged (verified)
Publish (`publish_commands.rs` `resolved_publish_path` passes nested paths
verbatim), `vault_sync.rs`, `vault_ref.rs` (already recursive/path-keyed), the TOML
`EntityStore`, the DB projection, faction folder logic, NPC/Item pickers (DB-based,
no vault-folder location picker), and Obsidian wikilinks.

## Accepted limitations
- Pre-existing flat notes stay flat; re-publishing a note whose folder changes
  (rename or kind edit) orphans the old `.md` (DB index is cleaned, file is not).
  Per the locked "no migration" decision, the GM sorts these by hand.
- Per decision, files are staged for the user's local review, never committed.
